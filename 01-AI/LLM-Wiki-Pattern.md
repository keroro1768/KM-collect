# LLM Wiki - 使用 LLM 建立個人知識庫的架構

> 來源：[llm-wiki.md](https://gist.githubusercontent.com/karpathy/442a6bf555914893e9891c11519de94f/raw/ac46de1ad27f92b28ac95459c782c07f6b8c964a/llm-wiki.md)
> 作者：Andrej Karpathy (@karpathy)
> 日期：2026年

## 🎯 核心概念

大多數人使用 LLM 和文檔的方式是 RAG：上傳檔案集合，LLM 在查詢時檢索相關片段並生成答案。這種方式有效，但 LLM 每個問題都要從頭發現知識。沒有累積。

**這個想法不同：**

不是只在查詢時從原始文檔檢索，LLM **增量建立和維護一個持久的 wiki**——一個結構化的、互相連結的 markdown 檔案集合，位於你和原始來源之間。

當你添加新來源時，LLM 不只索引它供後續檢索。它會閱讀它、提取關鍵資訊，並將其整合到現有 wiki 中——更新實體頁面、修訂主題摘要、註記新資料與舊聲明矛盾的地方、加強或挑戰不斷發展的綜合。知識編譯一次，然後*保持最新*，而不是每次查詢都重新推導。

**關鍵差異：wiki 是一個持久的、複合的-artifact。** 交叉引用已經存在。矛盾已經被標記。綜合已經反映了你所讀過的一切。wiki 隨著你添加的每個來源和每個問題變得越來越豐富。

## 🏗️ 架構

三層：

### 1️⃣ Raw Sources（原始來源）
- 你策展的來源文檔集合
- 文章、論文、圖片、資料檔
- 這些是不可變的——LLM 從中閱讀但從不修改
- 這是你的真理來源

### 2️⃣ The Wiki
- LLM 生成的 markdown 檔案目錄
- 摘要、實體頁面、概念頁面、比較、概述、綜合
- LLM 完全擁有這層
- 建立頁面、在新來源到來時更新它們、維護交叉引用、保持一切一致

### 3️⃣ The Schema
- 一份文件（例如 Claude Code 的 CLAUDE.md 或 Codex 的 AGENTS.md）
- 告訴 LLM wiki 是如何組織的、 convention 是什麼攝取來源、回答問題或維護 wiki 時要遵循什麼工作流程
- 這是關鍵配置文件——這是讓 LLM 成為有紀律的 wiki 維護者而不是通用聊天機器人的原因

---

## ⚙️ 操作

### Ingest（攝取）
你將新來源放入原始集合並告訴 LLM 處理它。

範例流程：
1. LLM 閱讀來源
2. 與你討論要點
3. 在 wiki 中寫摘要頁面
4. 更新索引
5. 更新 wiki 中相關的實體和概念頁面
6. 將條目追加到日誌

單一來源可能觸及 10-15 個 wiki 頁面。

### Query（查詢）
你針對 wiki 提問題。LLM 搜尋相關頁面、閱讀它們，並綜合帶有引用 的答案。

答案可以根據問題採取不同形式——markdown 頁面、比較表格、幻燈片（Marp）、圖表（matplotlib）、畫布。

**重要洞察：好的答案可以作為新頁面存回 wiki。** 你要求的比較、分析、你發現的連接——這些都是有價值的，不應該消失在聊天記錄中。

### Lint（整理）
定期要求 LLM 做健康檢查。尋找：
- 頁面之間的矛盾
- 被新來源取代的過時聲明
- 沒有入站連結的孤立頁面
- 提到但缺乏自己頁面的重要概念
- 缺失的交叉引用
- 可以用網路搜尋填補的資料缺口

## 📋 索引和日誌

兩個特殊檔案幫助 LLM（和你）導航不斷增長的 wiki：

### index.md（面向內容）
- wiki 中一切的目錄
- 每個頁面列出並帶有連結、一行摘要、可選元數據（如日期或來源數量）
- 按類別組織（實體、概念、來源等）
- LLM 在每次攝取時更新它
- ��回答查詢時，LLM 首先閱讀索引以找到相關頁面，然後深入研究它們

### log.md（時間順序）
- 發生什麼以及何時的追加-only 記錄
- 攝取、查詢、lint 穿透
- 有用的提示：如果每個條目以一致的前綴開頭（例如 `## [2026-04-02] ingest | Article Title`），日誌可以用簡單的 unix 工具解析

## 💡 技巧和竅門

| 技巧 | 說明 |
|------|------|
| **Obsidian Web Clipper** | 瀏覽器擴展將網路文章轉換為 markdown |
| **本地下載圖片** | 在 Obsidian 設定中設定固定的附件資料夾，然後使用熱鍵下載圖片 |
| **Obsidian 圖形視圖** | 查看 wiki 形狀的最佳方式——什麼連接到什麼 |
| **Marp** | 基於 markdown 的幻燈片格式，Obsidian 有插件 |
| **Dataview** | 在頁面 frontmatter 上運行查詢的 Obsidian 插件 |
| **Git** | wiki 就是一個 git repo——你免費獲得版本歷史、分支和協作 |

## 🌟 為什麼這有效

維護知識庫的繁瑣部分不是閱讀或思考——而是簿記。更新交叉引用、保持摘要是最新的、註記新資料何時與舊聲明矛盾、維持數十頁之間的一致性。

人類放棄 wiki 是因為維護負擔增長比價值快。LLM 不會無聊、不會忘記更新交叉引用、可以一次觸及 15 個檔案。wiki 保持維護因為維護成本接近零。

**人類的工作：** 策展來源、引導分析、問好問題、思考這一切意味著什麼。

**LLM 的工作：** 其他一切。
